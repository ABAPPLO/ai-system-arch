# 10 · 路线图与团队分工

| 阶段 | 时间 | 目标 | 产出 |

| Phase 0 | M1-M2 | 基础设施 | 云资源、K8s、CI/CD |

## 3. Phase 1：MVP（M3-M5）

### 3.1 目标
平台最核心能力可用：**注册接口、对外提供同步调用、记录调用日志、后台看监控**。

### 3.2 MVP 服务清单（精简到 6 个）

| 服务 | 是否做 | 说明 |
|------|--------|------|
| api-registry | ✅ | 接口管理（先 UI 接入，YAML 后续） |
| dispatcher | ✅ | 同步请求转发 |
| auth | ✅ | APIKey 鉴权 |
| quota | ✅（简化） | Redis 限流，不做配额结算 |
| docs | ✅（基础） | OpenAPI 生成 + curl/Python 示例 |
| admin-bff | ✅ | 后台前端 API |
| executor | ⏳ Phase 2 | 异步任务 |
| retry | ⏳ Phase 2 | |
| workflow-svc | ⏳ Phase 2 | |
| trace | ✅（基础） | ClickHouse 查询 |
| sdk-gen | ⏳ Phase 3 | |
| portal-bff | ⏳ Phase 3 | |
| audit | ✅（合并到 admin-bff） | |

### 3.3 关键任务

**后端**：
- 接口元数据 CRUD（api-registry）
- 接口发布流程（etcd → APISIX）
- APISIX 鉴权 / 限流插件配置
- dispatcher 同步转发 + Kafka 推送调用事件
- Kafka → ClickHouse 链路
- 调用日志查询 API
- 基础 OpenTelemetry 接入

**前端**：
- 接口列表 / 详情 / 新建 / 编辑
- 应用管理 / Key 生成
- 调用日志列表 / 详情
- 基础大盘

**SRE**：
- APISIX 配置
- ClickHouse 集群部署
- Kafka topic 创建
- 备份策略上线

### 3.4 里程碑
- M3 月底：单接口可注册、发布、被调用，日志可查
- M4 月底：完整 MVP 功能联调通过
- M5 月底：内部 1-2 个业务接口接入试运行

## 4. Phase 2：任务与韧性（M6-M8）

### 4.1 目标
**异步任务、失败重试、全链路追踪、灰度发布**。

### 4.2 任务

- executor 服务（异步任务）
- retry 服务（自动重试 + 后台手动）
- workflow-svc（Argo Workflow 封装）
- Jaeger 接入 + trace_id 全链路
- 失败任务后台 UI
- 灰度发布（Argo Rollouts）
- 声明式接入（YAML + CI）
- 接口评审工单
- 接口下线流程
- 告警规则上线（P0-P2）

### 4.3 里程碑
- M7 月底：异步任务 + 重试上线
- M8 月底：接口接入量 50+，调用 QPS 1w+

## 5. Phase 3：开放（M9-M11）

### 5.1 目标
**对外发布、自助接入、SDK、计费**。

### 5.2 任务

- 开发者门户 Portal 前端
- portal-bff 服务
- 外部用户注册 + 实名
- API 目录 / 搜索
- 在线调试
- SDK 自动生成（Python / Java / JS）
- 沙箱环境
- 配额与计费（quota 服务增强）
- 用量统计
- Webhook 通知
- 文档国际化

### 5.3 里程碑
- M10 月底：Portal Beta 上线，邀请 3-5 个外部调用方
- M11 月底：对外开放，接入 50+ 调用方

## 6. Phase 4：演进（M12+）

按需进行：

- AI 网关（LLM 统一接入 / 流式响应 / Token 计费）
- 多 Region 多活
- Go 重写热点服务（如 quota）
- 数据合规增强（GDPR）
- 高级分析（用户行为路径）
- API 市场化（计费、套餐、对账）

## 7. 团队分工（10+ 人）

### 7.1 推荐结构

```
平台负责人（1）
├── Platform 组（3）
│   ├── 网关 + 调度（1）
│   ├── 任务 + 重试（1）
│   └── 文档 + SDK（1）
├── Backend 组（4）
│   ├── 鉴权 + 配额（1）
│   ├── 调用日志 + 追踪（1）
│   ├── 后台 BFF（1）
│   └── 门户 BFF（1）
├── Frontend 组（2）
│   ├── Admin 前端（1）
│   └── Portal 前端（1）
├── SRE 组（2）
│   ├── 云资源 + K8s（1）
│   └── 监控 + 数据库（1）
└── QA 组（1-2）
    └── 测试 + 自动化
```

合计：12-13 人。

### 7.2 各组职责

**Platform 组**：
- 平台核心架构（dispatcher、executor、retry、workflow）
- 关键技术决策
- 跨组技术对齐

**Backend 组**：
- 业务服务开发
- 与 Platform 组接口对齐
- 接入业务方协助

**Frontend 组**：
- Admin / Portal 全栈
- 与 Backend BFF 对齐
- UX / UI 落地

**SRE 组**：
- 云资源 + K8s
- 监控 + 告警
- 故障响应
- 成本优化

**QA 组**：
- 自动化测试
- 压测
- 上线验证

### 7.3 工作模式

- 双周迭代（敏捷）
- 每日站会（按组）
- 每月跨组技术评审
- 每季度回顾 + 路线图调整

### 7.4 招聘画像

| 角色 | 关键技能 |
|------|---------|
| Platform | Python async、K8s、Kafka、APISIX、高并发设计 |
| Backend | FastAPI、PostgreSQL、Redis、分布式系统 |
| Frontend | Vue 3、TypeScript、大型后台经验 |

## 8. 风险与对策

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| Python 性能瓶颈 | 中 | 高 | 早期压测、必要时 Go 重写热点 |
| 接口接入积极性低 | 高 | 高 | MVP 期间拉 1-2 个核心业务试点 |
| 外部开发者不感兴趣 | 中 | 中 | 优先接入已有合作方 |
| 团队招聘困难 | 高 | 高 | 提前 2 个月启动、内部转岗 |
| 阿里云故障 | 低 | 高 | 跨 AZ + 备份 + SOP |
| 成本超预算 | 中 | 中 | Phase 1 简化规模、容量监控 |
| 需求蔓延 | 高 | 中 | 严格 Phase 范围、新需求入下一期 |
| 安全事件 | 低 | 极高 | 上线前渗透测试、KMS 加密、审计 |
