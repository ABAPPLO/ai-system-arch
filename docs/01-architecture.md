# 01 · 整体架构

## 1. 设计原则

| 原则 | 含义 |
|------|------|
| **API 即资产** | 接口有完整生命周期：设计 → 注册 → 发布 → 监控 → 下线 |
| **声明式优先** | 接口定义走 YAML + Git，可评审、可回滚、可 GitOps |
| **平台不绑业务** | 平台只做横切关注点（鉴权 / 限流 / 日志 / 重试），业务逻辑由业务方实现 |
| **统一契约** | 同步 / 异步 / 工作流任务对外契约一致，调用方零感知 |
| **失败是常态** | 任何失败可追溯、可重试、可补偿 |
| **水平扩展** | 服务无状态，数据按 partition 分片 |
| **多租户隔离** | 所有数据带 tenant_id，应用层 + RLS 双重隔离（[ADR-009](00-decisions.md#adr-009-多租户策略)） |
| **AI 友好** | 元数据 / 日志 / 配额为 LLM 流式调用预留扩展（[ADR-004](00-decisions.md#adr-004-ai-网关扩展)） |
| **安全默认开** | 默认 HTTPS、默认鉴权、默认限流、默认脱敏、默认审计（[ADR-010 等保 2.0 三级](00-decisions.md#adr-010-数据合规)） |

## 3. 各层职责

### 3.1 接入层
- **DNS**：主备双线解析，容灾切换
- **CDN**：静态资源（前端 + 文档）回源率 < 5%
- **WAF**：SQL/XSS/CC 防护，规则 + ML 双引擎
- **SLB**：4 层负载，流量入口
- **Nginx**：7 层负载，SSL 终止，基础流量治理

### 3.2 网关层（APISIX）
**核心职责**：
- 动态路由（基于 path / header / query）
- 鉴权（JWT / APIKey / HMAC 签名 / OAuth2），从 API Key 反查 tenant_id
- 多维限流（**租户 / 调用方 / API / 全局**，秒 / 分 / 天）—— 见 [ADR-009](00-decisions.md#adr-009-多租户策略)
- 熔断、降级
- 灰度发布（按租户 / 调用方 / Header / 比例）
- 请求 / 响应转换
- 调用日志异步推 Kafka（APISIX kafka-plugin），含 tenant_id / token_usage（AI 接口）
- 链路追踪注入（生成 `X-Trace-Id` + `X-Tenant-Id`）

**性能预期**：单节点 5w+ QPS，集群水平扩展。

### 3.3 业务层
见 [03-services.md](03-services.md)。

### 3.4 中间件层
| 组件 | 用途 | 规模 |
|------|------|------|
| Kafka | 调用事件流、任务消息、CDC | 3-6 broker，10w+ QPS |
| Redis Cluster | 缓存、限流计数、分布式锁、延迟队列 | 6-30 节点 |
| etcd | APISIX 配置存储、服务发现 | 3-5 节点 |
| MinIO | 请求大 body、错误堆栈、SDK 包 | 多副本 |

### 3.5 存储层
| 组件 | 用途 | 规模 |
|------|------|------|
| PostgreSQL | 元数据（接口 / 调用方 / 授权 / 配额） | 主备 + 只读 |
| ClickHouse | 调用日志、指标聚合 | 集群（shard + replica） |
| Jaeger (Cassandra / ES) | 链路 Span | 集群 |

## 5. 关键设计权衡

### 5.1 为什么选 APISIX 而非自研网关 / Kong
- ✅ 性能：Nginx + etcd，实测 5w+ QPS / 节点
- ✅ 动态配置：不需要 reload，改路由秒级生效
- ✅ 插件生态：鉴权、限流、Kafka 推送、trace 都开箱即用
- ✅ 中文社区：文档好，踩坑少
- ❌ 自研成本：Lua/OpenResty 团队不好招；Kong 性能略逊

### 5.2 为什么调用日志走 ClickHouse 而非 ES
| 维度 | ClickHouse | Elasticsearch |
|-----|-----------|---------------|
| 写入吞吐 | 100w+/s | 10w/s |
| 存储压缩 | ~10x | ~1x |
| 聚合查询 | 极快 | 一般 |
| 全文检索 | 一般 | 极强 |
| 成本 | 低 | 高（约 10x） |

**结论**：调用日志主存 ClickHouse；只在错误堆栈全文字段补一个 ES 索引（可选）。

### 5.3 Python 应对 10w QPS 是否合理
- 网关层 APISIX 不是 Python，核心路由 / 限流 / 鉴权在 Nginx 级别
- 业务层 QPS 经网关分摊，单业务服务 QPS 1k-1w，Python async 完全可承担
- 极端性能瓶颈的微服务（如配额计数、限流决策）可后续改 Go，平台层不依赖 Python 单点
- 详见 [06-high-concurrency.md](06-high-concurrency.md)

### 5.4 为什么用 PostgreSQL 而非 MySQL
- JSONB 字段直接存 JSON Schema，可索引、可查询
- DDL 事务、CTE、窗口函数支持好
- 后续要做时序 / 地理等扩展，PG 生态更全
- 阿里云 RDS PG 性价比与 MySQL 持平

### 5.5 为什么 Kafka + Argo Workflow 双轨
- Kafka 适合海量短任务（秒~分钟级），流式消费
- Argo Workflow 适合复杂 DAG 长任务（分钟~小时级），支持依赖、重试、断点
- 两者职责不冲突，共同覆盖「短 / 长」两类
- 不用 Celery 的原因：Celery 在 10w QPS 场景下 broker 压力大，可观测性弱

### 5.6 前后端如何切分
- **后台 BFF（Admin BFF）**：服务内部用户，功能复杂，权限严格
- **门户 BFF（Portal BFF）**：服务外部开发者，功能聚焦，开放性强
- 两套前端独立部署，各自 Vue 3

### 5.7 为什么选平台多租户（[ADR-009](00-decisions.md#adr-009-多租户策略)）
- 公司多个业务线 / 子公司需要数据隔离、配额独立、报表独立
- 不选商业多租户：当前无 SaaS 卖给外部企业作为独立产品的需求
- 不选单租户：所有业务线混在一起，无法独立治理
- 详见 [11-multi-tenant.md](11-multi-tenant.md)
- **代价**：所有表加 `tenant_id`、所有查询带租户过滤、所有缓存 key 加租户前缀；但 RLS 兜底防止应用层 bug

### 5.8 为什么现在就预留 AI 网关（[ADR-004](00-decisions.md#adr-004-ai-网关扩展)）
- Python 选型的战略价值就在 AI 生态
- 预留成本极低（仅 schema 加字段、文档生成支持流式）
- 不预留的代价：Phase 4 改造涉及元数据、调用日志、配额、文档多个模块
- 预留≠实现：Phase 1/2 不实际接入 LLM，仅 schema 层支持

### 5.7 为什么不直接把 PG 当调用日志库
- 写入量 10w/s 直接打挂 PG
- 调用日志主要场景是「按 API / 时间 / 状态聚合」，OLAP 适合 ClickHouse
- PG 只存元数据，调用日志走 ClickHouse，职责清晰

## 6. 与外部系统的边界

| 方向 | 协议 | 说明 |
|------|------|------|
| 调用方 → 网关 | HTTPS / REST | 标准 HTTP/JSON |
| 调用方 → 网关 | gRPC（可选） | 高性能场景预留 |
| 网关 → 业务服务 | HTTP / gRPC | K8s 内部 |
| 业务服务 → 后端实现 | HTTP / gRPC / MQ | 业务方自由选择 |
| 平台 → 业务方 Webhook | HTTPS POST | 异步任务结果回调 |
| 平台 → 监控告警 | Webhook | 钉钉 / 飞书 / 企微 |
| 平台 → BI | ClickHouse 直查 / JDBC | 报表分析 |

## 7. 关键非功能需求

| 指标 | 目标 |
|------|------|
| 平台整体 QPS | ≥ 10w |
| 同步接口 P99 延迟 | < 200ms（不含业务后端） |
| 异步任务调度延迟 | < 1s（提交到执行） |
| 月度可用性 SLA | 99.95% |
| 调用日志写入成功率 | ≥ 99.99% |
| 调用日志查询 P95 | < 1s（单 trace_id）/ < 5s（聚合） |
| 重试触发延迟 | < 30s |
| 接口发布生效 | < 10s |
