# 05 · 核心流程

## 1. 接口接入与发布流程

### 1.1 声明式接入（推荐）

```
开发者 → Git Repo (api.yaml) → GitLab CI →  校验 →  PR → 评审 → 合并
                                                                  ↓
                                                          ArgoCD 监听
                                                                  ↓
                                                  api-registry 写入元数据
                                                                  ↓
                                                       docs / sdk-gen 触发
                                                                  ↓
                                                    APISIX 路由自动下发
                                                                  ↓
                                                       dev 环境可用
```

**api.yaml 示例**：

```yaml
apiVersion: apihub.io/v1
kind: APIDefinition
metadata:
  name: user.create
  owner: user-service-team
  tags: [user, core]
spec:
  path: /v1/users
  method: POST
  mode: sync
  auth:
    type: jwt
  rateLimit:
    qps: 100
    qpd: 100000
  timeout: 3000ms
  retry:
    auto: true
    max: 3
    backoff: exponential
  backend:
    type: http
    target: http://user-service.internal:8080/internal/users
    transform:
      jq: '. + {source: "apihub"}'
  requestSchema:
    type: object
    required: [name]
    properties:
      name: { type: string, minLength: 1, maxLength: 64 }
      age:  { type: integer, minimum: 0, maximum: 150 }
  responseSchema:
    type: object
    properties:
      id:   { type: string }
      name: { type: string }
      age:  { type: integer }
  examples:
    - request: { name: "alice", age: 18 }
      response: { id: "u_001", name: "alice", age: 18 }
```

### 1.2 UI 接入

```
开发者 → 后台新建接口 → 填表单 / 上传 Schema → 提交评审
                                                          ↓
                                                  审批通过
                                                          ↓
                                              一键发布到 dev
                                                          ↓
                                          dev 自测通过 → 晋升 staging
                                                          ↓
                                          staging 验收 → 晋升 prod（灰度比例）
```

### 1.3 详细发布流程（按环境分级审批，[ADR-005](00-decisions.md#adr-005-审批流强度)）

```
[提交发布]
       │
       ├─ 目标环境 = dev？
       │     └─ 自助发布：直接走下面流程
       │
       ├─ 目标环境 = staging？
       │     └─ 简单审批：业务负责人钉钉确认 → 走下面流程
       │
       └─ 目标环境 = prod？
             └─ 强审批：钉钉审批流（运维 + 业务负责人）→ 通过后走下面流程

[审批通过]
       │
       ▼
┌──────────────────┐
│  api-registry    │  1. 校验 Schema 合法性
│                  │  2. 创建新版本记录（带 tenant_id）
│                  │  3. 写 etcd（APISIX 路由）
│                  │  4. 失效 Redis 缓存（t:{tenant_id}:api:*）
│                  │  5. 触发 docs/sdk-gen
└──────────────────┘
       │
       ▼
┌──────────────────┐
│  APISIX          │  watch etcd，秒级生效
└──────────────────┘
       │
       ▼
┌──────────────────┐
│  audit           │  记录变更（带 tenant_id, actor, auth_method）
└──────────────────┘
       │
       ▼
┌──────────────────┐
│  notification    │  Webhook / 钉钉工作通知 → 受影响调用方
└──────────────────┘
```

**生效时间目标**：< 10s（审批通过后到路由可用）。

**审批流详情**：
- dev：无审批，接口提供方自助
- staging：钉钉机器人推送到业务群，负责人点"同意"即可
- prod：钉钉审批流（多级），含灰度策略，审批通过后按 5% → 25% → 50% → 100% 自动推进

## 2. 同步调用流程

```
调用方                APISIX              auth           quota           dispatcher         业务后端
  │                     │                  │              │                  │                  │
  │  POST /v1/users     │                  │              │                  │                  │
  │  Authorization: ... │                  │              │                  │                  │
  │  X-Trace-Id: ...    │                  │              │                  │                  │
  ├────────────────────▶│                  │              │                  │                  │
  │                     │ 1. 注入 trace     │              │                  │                  │
  │                     │ 2. 路由匹配       │              │                  │                  │
  │                     │ 3. 鉴权           │              │                  │                  │
  │                     ├─────────────────▶│              │                  │                  │
  │                     │                  │ 校验 Key      │                  │                  │
  │                     │                  │ 校验签名      │                  │                  │
  │                     │                  │ 查授权关系    │                  │                  │
  │                     │◀─────────────────┤              │                  │                  │
  │                     │ 4. 限流决策       │              │                  │                  │
  │                     ├─────────────────────────────────▶│                  │                  │
  │                     │                  │              │ Redis INCR       │                  │
  │                     │                  │              │ 判断超额          │                  │
  │                     │◀─────────────────────────────────┤                  │                  │
  │                     │ 5. 转发           │              │                  │                  │
  │                     ├──────────────────────────────────────────────────▶│                  │
  │                     │                  │              │                  │ 6. 调用业务后端   │
  │                     │                  │              │                  ├─────────────────▶│
  │                     │                  │              │                  │                  │ 处理
  │                     │                  │              │                  │◀─────────────────┤
  │                     │                  │              │                  │ 7. 包装响应      │
  │                     │◀──────────────────────────────────────────────────┤                  │
  │                     │ 8. 推 Kafka 日志  │              │                  │                  │
  │                     │ 9. 配额计数       │              │                  │                  │
  │                     ├─────────────────────────────────▶│                  │                  │
  │                     │ 10. 返回响应      │              │                  │                  │
  │◀────────────────────┤                  │              │                  │                  │
  │                     │                  │              │                  │                  │
```

**关键点**：
- trace_id 在网关生成（如果调用方未传），全程透传
- **tenant_id 由网关从 API Key 反查**，注入 `X-Tenant-Id` Header，下游服务透传（[ADR-009](00-decisions.md#adr-009-多租户策略)）
- 三层配额检查：租户总配额 / 应用配额 / API 限流
- 鉴权结果缓存 1-10 分钟（带 tenant_id），避免每次打 auth 服务
- 限流决策 < 1ms（Redis 本地操作）
- 调用事件异步推 Kafka，header 带 tenant_id + trace_id，不阻塞主链路

## 3. 异步短时任务流程

```
调用方                 APISIX            dispatcher         Kafka           executor         业务后端
  │                      │                  │                │                 │                  │
  │  POST /v1/users      │                  │                │                 │                  │
  │  X-Mode: async       │                  │                │                 │                  │
  ├─────────────────────▶│                  │                │                 │                  │
  │                      │                  │                │                 │                  │
  │                      ├─────────────────▶│                │                 │                  │
  │                      │                  │ 1. 生成 task_id │                 │                  │
  │                      │                  │ 2. 写 PG        │                 │                  │
  │                      │                  │ status=pending  │                 │                  │
  │                      │                  │ 3. 推 Kafka     │                 │                  │
  │                      │                  ├───────────────▶│                 │                  │
  │                      │                  │                │                 │                  │
  │  返回 task_id        │                  │                │                 │                  │
  │◀─────────────────────┤                  │                │                 │                  │
  │                      │                  │                │                 │                  │
  │                                                                                                  │
  │  （调用方轮询 /task/{id}，或等 Webhook）                                                          │
  │                                                                                                  │
  │                      │                  │                │ 4. 消费         │                  │
  │                      │                  │                ├────────────────▶│                  │
  │                      │                  │                │                 │ status=running   │
  │                      │                  │                │                 │ 调用业务后端      │
  │                      │                  │                │                 ├─────────────────▶│
  │                      │                  │                │                 │                  │ 处理
  │                      │                  │                │                 │◀─────────────────┤
  │                      │                  │                │                 │ status=succeeded │
  │                      │                  │                │                 │ 写 PG / Webhook  │
  │                      │                  │                │                 │                  │
```

**关键点**：
- 调用方立即拿到 task_id，不等处理
- 业务处理在 executor worker 中，dispatcher 不阻塞
- 结果回写 PG + 触发 Webhook（带签名）
- 调用方可 `GET /tasks/{task_id}` 轮询查询

## 4. 长时工作流流程

```
调用方              dispatcher          workflow-svc        Argo Workflow          MinIO
  │                    │                    │                    │                    │
  │  POST /v1/jobs     │                    │                    │                    │
  │  mode: workflow    │                    │                    │                    │
  ├───────────────────▶│                    │                    │                    │
  │                    ├───────────────────▶│                    │                    │
  │                    │                    │ 1. 提交 Argo WF     │                    │
  │                    │                    ├───────────────────▶│                    │
  │                    │                    │                    │ 启动 Pod           │
  │                    │                    │                    │ 执行 step1         │
  │                    │                    │                    │ 执行 step2         │
  │                    │                    │                    │ ...                │
  │                    │                    │                    │ 产物上传           │
  │                    │                    │                    ├───────────────────▶│
  │                    │                    │ 2. 监听状态         │                    │
  │                    │                    │◀───────────────────┤                    │
  │                    │                    │                    │                    │
  │  GET /jobs/{id}    │                    │                    │                    │
  ├───────────────────▶│                    │                    │                    │
  │  返回进度 + 步骤    │                    │                    │                    │
  │◀───────────────────┤                    │                    │                    │
```

**关键点**：
- Argo Workflow 跑在独立 namespace，资源配额隔离
- 长任务支持中断、恢复、取消
- 每一步的状态、日志、产物都持久化到 MinIO
- 超时由 workflow-svc 控制，可按业务配置

## 4.5 AI 流式调用流程（[ADR-004 预留](00-decisions.md#adr-004-ai-网关扩展)，Phase 4 实现）

```
调用方              APISIX          dispatcher         LLM Provider (OpenAI/Anthropic)
  │                   │                 │                    │
  │  POST /v1/chat    │                 │                    │
  │  Accept: text/    │                 │                    │
  │  event-stream     │                 │                    │
  ├──────────────────▶│                 │                    │
  │                   │ 鉴权 + 限流       │                    │
  │                   ├────────────────▶│                    │
  │                   │                 │ 1. 识别 ai_model   │
  │                   │                 │ 2. 取 API Key      │
  │                   │                 │ 3. 拼接 prompt     │
  │                   │                 │ 4. 流式调用         │
  │                   │                 ├───────────────────▶│
  │                   │                 │                    │ 流式返回 chunk
  │                   │                 │◀───── chunk 1 ─────┤
  │                   │◀──── chunk 1 ───┤                    │
  │◀── data: {...} ───┤                 │                    │
  │                   │                 │◀───── chunk 2 ─────┤
  │                   │◀──── chunk 2 ───┤                    │
  │◀── data: {...} ───┤                 │                    │
  │                   │                 │       ...          │
  │                   │                 │◀───── [DONE] ──────┤
  │                   │                 │ 5. 统计 token_total │
  │                   │                 │ 6. 推 Kafka 日志    │
  │                   │                 │    含 token_usage   │
  │                   │                 │ 7. 配额扣减 token   │
  │◀── [DONE] ────────┤                 │                    │
```

**关键点**：
- SSE 协议（`text/event-stream`），每个 chunk 立即转发
- 流式期间保持连接，超时单独控制（默认 5 分钟）
- Token 用量在流结束时一次性统计，推 Kafka
- 配额按 token 数扣减（不是按调用次数）
- 流式中断（用户断开 / 网络异常）按已生成 token 计费

## 5. 失败与重试流程

### 5.1 三道防线

```
调用失败
    │
    ▼
┌──────────────────────────────────┐
│ 第 1 道：网关层自动重试            │
│ - GET / 幂等 POST                 │
│ - 立即重试 1-2 次                  │
│ - 仅针对 5xx / 网络错误            │
└──────────────────────────────────┘
    │ 仍失败
    ▼
┌──────────────────────────────────┐
│ 第 2 道：业务层自动重试（retry）   │
│ - 指数退避：1s/4s/16s/64s         │
│ - 最大次数（默认 3-5）             │
│ - 仅对幂等接口                     │
│ - 通过 Kafka + Redis ZSet          │
└──────────────────────────────────┘
    │ 仍失败
    ▼
┌──────────────────────────────────┐
│ 第 3 道：后台手动重试              │
│ - 入死信队列                       │
│ - 后台展示完整错误                 │
│ - 用户判断后触发                   │
└──────────────────────────────────┘
```

### 5.2 自动重试详细流程

```
executor 调用业务后端失败
        │
        ▼
┌─────────────────────────┐
│ executor 写 PG          │
│ retry_task 记录          │
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│ 推 Kafka                │
│ topic=task-failures     │
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│ retry 服务消费           │
│ 按策略计算下次重试时间    │
│ 写 Redis ZSet           │
│ score = next_retry_at   │
└────────────┬────────────┘
             │
             ▼
       ... 等待 ...
             │
             ▼
┌─────────────────────────┐
│ retry worker 扫描        │
│ ZSet 到期任务            │
└────────────┬────────────┘
             │
             ▼
┌─────────────────────────┐
│ 调用 executor 重试       │
│ retry_no + 1             │
└────────────┬────────────┘
             │
        ┌────┴────┐
        │         │
     成功      失败
        │         │
        ▼         ▼
┌──────────┐  ┌────────────────────┐
│ 写 PG    │  │ retry_count++       │
│ 关闭记录 │  │ 判断 < max?         │
│ 回调调用 │  │  是 → 再写 ZSet     │
│ 方       │  │  否 → 死信队列      │
└──────────┘  └────────────────────┘
```

### 5.4 重试幂等性保证

- **必须幂等的接口才允许重试**：API 元数据中 `idempotent: true` 声明
- 重试时调用方应携带 `Idempotency-Key`，业务方据此去重
- 平台强制：所有 POST/PUT/DELETE 默认 `idempotent: false`，需显式开启重试

    ▼

    ▼

**保护机制**：

## 8. 灰度发布流程

新接口版本上线时：

```
v1 100% 流量
        │
        ▼
v2 发布，初始 0%
        │
        ▼
v2 按 1% → 10% → 50% → 100% 灰度
        │
        ▼
每阶段观察：
- 错误率
- 延迟
- 调用方反馈
        │
        ▼
问题 → 立即回滚到 v1
无问题 → 全量到 v2，下线 v1
```

灰度维度：
- **按调用方**：指定应用先走 v2
- **按 Header**：`X-Canary: true`
- **按比例**：随机百分比
- **按地理位置**：仅对某地区（可选）

## 9. 关键 SLA 与告警

| 指标 | 阈值 | 动作 |
|------|------|------|
| 网关 5xx 错误率 | > 1%（5min） | 告警值班 |
| 接口 P99 延迟 | > 1s（5min） | 告警 |
| 调用日志写入失败率 | > 0.1% | 告警 + 降级 |
| 重试积压 | > 1000 待处理 | 告警 |
| 死信队列新增 | > 10 / 分钟 | 告警 |
| Kafka 消费延迟 | > 30s | 告警 |
| Redis 内存 | > 80% | 告警 |
| ClickHouse 写入失败 | > 0.01% | 告警值班 |
| 证书到期 | < 14 天 | 告警 |

## 10. 灾备与故障切换

| 故障场景 | 切换策略 |
|---------|---------|
| 单 Pod 挂 | K8s 自动重启 |
| 单可用区挂 | SLB 自动切到其他 AZ |
| PG 主挂 | 自动切备（RDS 高可用） |
| Redis 单节点挂 | Cluster 自动 failover |
| Kafka broker 挂 | 副本自动选主 |
| APISIX 节点挂 | 多副本 + 健康检查 |
| 业务服务全挂 | 降级：返回缓存 / 静态响应 |
| 整 Region 挂 | 跨 Region DNS 切换（RTO 30min） |

RTO / RPO 目标：
- **RTO**：核心功能 30 分钟内恢复
- **RPO**：调用日志 1 分钟内（Kafka 副本）；元数据 0（PG 同步复制）
